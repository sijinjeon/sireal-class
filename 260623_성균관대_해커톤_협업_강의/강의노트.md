# 해커톤 협업 강의 노트

> 전시진 (SIREAL) · 성균관대학교 해커톤 협업 강의

## 인트로

이 문서는 **개발자와 비개발자가 섞인 팀이 AI와 함께 해커톤 프로젝트를 완성**하기 위한 협업 가이드다.

핵심은 두 가지다.

- **이제는 직접 만드는 게 아니라 AI에게 "잘 시키는" 시대다.** 잘 시키려면 명확한 기록과 문서가 필요하다.
- **모든 대화·자료·결정은 노션에 기록으로 남긴다.** 그 기록이 곧 AI가 일하는 재료(컨텍스트)가 된다.

소통 창구 만들기부터 발표까지, 아래 흐름을 순서대로 따라간다.

## 전체 흐름

```
소통 창구 → 아이데이션 → 아이디어 선별 → 문서화 → 구현 → 오류수정 → 발표준비 → 발표
```

각 단계는 아래 **5요소를 같은 순서**로 설명한다.
`방법·순서 → 도구 → 규칙 → 실습 → 점검`

-----

## 1. 소통 창구 만들기

**방법·순서** — 노션 페이지 생성 → 팀원 초대 → 페이지 안에 데이터베이스 생성 → 각종 문서를 종류별로 정리

**도구** — Notion (+ 나중에 MCP 연결)

**규칙**

1. **매 섹션·회의를 시작할 때마다 항상 AI 회의 노트를 켠다.** 모든 대화 내용을 빠짐없이 기록으로 남기는 것이 가장 중요하다.
1. 모든 자료·기록은 노션 한곳에 모은다.
1. 제목은 알아보기 쉽게 짓는다.
1. 데이터베이스에는 `종류` 속성(select) 하나만 달아 분류한다. (회의록 / 자료 / 할 일 / 문서 등)
1. 좋은 의견과 결정은 반드시 노션에 문서로 남긴다.

> ⚠️ 회의는 무조건 "AI 노트 켜기"부터. 켜지 않은 회의는 흔적이 사라진다.
>
> 복잡한 DB 구조는 필요 없다. `종류` 속성 하나면 나중에 MCP로 "회의록만 모아줘" 같은 활용이 가능해진다.

> **Q. 해커톤 하느라 바빠 죽겠는데 소통 창구 만들 시간이 어딨어요?**
>
> 바쁠수록 더 필요하다. 소통 창구가 없으면 "아까 누가 뭐 말했더라"를 반복하다 시간을 다 쓴다. 노션 페이지 만들고 팀원 초대하는 건 5분이다. 그 5분이 이후 문서화 → AI 구현 → 발표준비까지 전부 연결되는 기반이 된다.

**실습** — 노션 페이지 만들고 팀원 초대 → DB 생성 → 첫 항목(회의록 등) 올려보기

**점검** — 전원 초대됐는지, 문서가 한곳에 종류별로 쌓이는지 확인

-----

## 2. 아이데이션

**방법·순서** — 문제 정의 → 자료수집 → 회의로 아이디어 쏟아내기

**도구** — 노션 AI 노트(우선) / 클로바노트(대안), AI 음성 모드, 검색

- 회의록은 **노션 AI 노트**로 작성하는 것을 우선한다.
- 팀원 전체가 노션 AI 노트를 쓸 수 없으면 **클로바노트**(웹 접속 가능)를 사용한다.

**규칙**

- **회의 시작과 동시에 AI 노트부터 켠다.**
- 다른 사람의 아이디어를 비판하지 않는다.
- 처음 아이데이션 단계에서는 반드시 긍정적으로, 아이디어를 더 확장시킬 수 있는 아이디어만 내놓는다.
- 판단은 보류하고 양으로 승부한다.
- 떠오르면 즉시 기록한다.

**실습**

1. 기본은 **클로바노트**로 아이디어 회의를 진행하고, 그 회의 내용을 문서로 정리한다.
1. **노션 AI 노트**로도 동일하게 실습할 수 있다.

**점검** — 정리된 문서를 공유 → 흥미도·실현성 코멘트

-----

## 3. 아이디어 선별

**방법·순서** — 후보 나열 → 심사 기준 적용 → 우선순위 → 1개 확정

**심사 기준 5가지**

1. 문제 정의 및 사회적 필요성
1. 창의성 및 독창성
1. AI 활용의 적절성 및 기술 타당성
1. 실현 가능성 및 완성도
1. 기대 효과 및 파급력

**도구** — 평가표(Notion/스프레드시트), AI에게 기준별 평가 요청

**규칙**

- 심사위원 경험 및 기준으로 본다.
- "이거 좋은데요?" 반응을 신호로 포착한다.
- 정한 이유를 문서에 써둔다.

**실습** — 심사 기준표를 만들고 후보를 점수화 → 1개 선택

**점검** — 선정 사유 발표 → 기준이 명확한지 확인

-----

## 4. 문서화

**방법·순서** — 문서 목록 잡기 → 마크다운 작성 → 검토 → 하나의 폴더로 묶기 → 지속 업데이트

**도구** — 마크다운 에디터, Claude, Cursor

**규칙**

1. AI에게 너무 의존하면 문장만 늘어난다.
1. 마크다운으로 제작 → AI가 가장 좋아하는 문서 형식
1. 마크다운은 최소 2,000줄 이내로 만들기 → 짧을수록 좋음
1. 문서가 너무 길어지면 주제별로 문서 쪼개기
1. 문서는 하나의 폴더에 묶어두기
1. 의사결정이 필요 → 경험이 많다고 모든 결정이 좋을 수는 없다.

> AI도 결정을 대신 안 한다. 프롬프트를 입력하는 사람의 기반에 맞춰준다.

**실습** — 기능명세(PRD)를 마크다운 1장으로 작성

**점검** — "AI가 읽기 좋은 구조인가" 상호 리뷰

-----

## 5. 구현

**방법·순서** — 문서 기반으로 AI에게 방향 지시 → Plan mode로 계획 → Agent mode로 구현 → Repo·배포 선정

- **Plan mode**, **Agent mode**
- **GitHub**, **Vercel**
- Repo 선정 → 배포 선정 → 언어 구현한 뒤에 오류수정해도 늦지 않다.

**도구** — Cursor / Claude Code, GitHub, Vercel

**규칙 — 잘 시키는 법 (지시문 작성)**

이제는 직접 만드는 게 아니라 **AI에게 시키는 시대**다. 잘 시키려면 지시문(프롬프트)을 잘 써야 한다.

1. **구체적으로** — 모호한 표현 대신 원하는 결과를 명확히 적는다.
1. **맥락을 함께** — 관련 문서·파일을 근거로 주고 무엇을 만들지 알려준다.
1. **한 번에 하나씩** — 큰 작업은 작은 단위로 쪼개서 지시한다.
1. **제약을 명시** — 언어·라이브러리·스타일·형식 등 조건을 분명히 한다.
1. **기대 결과를 보여준다** — 예시나 원하는 출력 형태를 함께 제시한다.
1. **안 되면 다시** — 결과를 확인하고 부족하면 지시문을 고쳐 반복한다.

> 결국 결과물의 품질은 **지시문의 품질**이 결정한다.

**실습** — 지시문 1개를 작성해 첫 기능 구현하기

**점검** — 지시문 ↔ 결과 비교 → 더 나은 지시문으로 개선

-----

## 6. 오류수정

**방법·순서** — 무조건 오류는 생긴다. 오류 재현 → **Test + 모니터링** 반복 → 수정 → 확인되면 Commit + Push

**도구** — 브라우저 개발자도구, Git, Changelog / README

**규칙**

1. 수정해서 내가 원하는 것이 구현됐을 때 **Commit + Push**
1. Browser에서 계속 실시간 확인하기
1. 수정된 모듈은 **Changelog**와 **README** 파일에 기록 남기기 → 발표자료로 활용
1. 기능을 추가하고 싶을 때는 **New 세션** 열어서 진행

**실습** — 버그 1개를 고치고 커밋해 보기

**점검** — 커밋 메시지·기록 확인

-----

## 7. 발표준비

**방법·순서** — 목차 → 내용 → 시연하며 슬라이드 생성

**도구** — Gemini, NotebookLM, Claude, Claude Design

**규칙**

- 문서는 이미 만들어져 있다. 구현·시연은 재미있게 만들어가는 게 낫다.
- 보통 발표 내용은 AI로 만들면 된다.
- 슬라이드 생성은 쉽다. 안에 들어가는 내용이 항상 어렵다.
- 한 번에 모든 슬라이드를 만드는 것보다 목차 → 내용 → 시연하며 생성.

**실습** — 문서 기반으로 3장짜리 슬라이드 초안 생성

**점검** — 모의 발표 → 시간·전달력 확인
