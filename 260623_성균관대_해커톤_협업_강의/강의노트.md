# 해커톤 협업 강의 노트

> 전시진 (SIREAL) · 성균관대학교 해커톤 협업 강의

## 인트로

이 문서는 **개발자와 비개발자가 섞인 팀이 AI와 함께 해커톤 프로젝트를 완성**하기 위한 협업 가이드다.

핵심은 두 가지다.

- **이제는 직접 만드는 게 아니라 AI에게 "잘 시키는" 시대다.** 잘 시키려면 명확한 기록과 문서가 필요하다.
- **모든 대화·자료·결정은 노션에 기록으로 남긴다.** 그 기록이 곧 AI가 일하는 재료(컨텍스트)가 된다.

소통 창구 만들기부터 발표 준비까지, 아래 흐름을 순서대로 따라간다.

## 전체 흐름

```
소통 창구 → 아이데이션 ↔ 아이디어 선별 → 문서화 → 구현 → 오류수정 → 발표준비
```

아이데이션과 아이디어 선별은 순서가 아니라 **반복 루프**다. 괜찮은 아이디어가 나오면 선별 단계로 넘겨 검증하고, 결과를 보면서 다시 아이디어를 발전시킨다.

각 단계는 아래 **5요소를 같은 순서**로 설명한다.
`방법·순서 → 도구 → 규칙 → 실습 → 점검`

---

## 1. 소통 창구 만들기

**방법·순서** — 노션 페이지 생성 → 팀원 초대 → 페이지 안에 데이터베이스 생성 → 각종 문서를 종류별로 정리

**도구** — Notion (+ 나중에 MCP 연결)

**규칙**

1. **매 섹션·회의를 시작할 때마다 항상 AI 회의 노트를 켠다.** 모든 대화 내용을 빠짐없이 기록으로 남기는 것이 가장 중요하다.
1. 모든 자료·기록은 노션 한곳에 모은다.
1. 제목은 알아보기 쉽게 짓는다.
1. 데이터베이스에는 `종류` 속성(select) 하나만 달아 분류한다. (회의록 / 자료 / 할 일 / 문서 등)
1. 좋은 의견과 결정은 반드시 노션에 문서로 남긴다.

> ⚠️ 회의는 무조건 "AI 노트 켜기"부터. 켜지 않은 회의는 흔적이 사라진다.
>
> 복잡한 DB 구조는 필요 없다. `종류` 속성 하나면 나중에 MCP로 "회의록만 모아줘" 같은 활용이 가능해진다.

**실습** — 노션 페이지 만들고 팀원 초대 → DB 생성 → 첫 항목(회의록 등) 올려보기

**점검** — 전원 초대됐는지, 문서가 한곳에 종류별로 쌓이는지 확인

---

## 2. 아이데이션

**방법·순서** — 문제 정의 → 자료수집 → 회의로 아이디어 쏟아내기

**도구** — 노션 AI 노트(우선) / 클로바노트(대안), AI 음성 모드, 검색

- 회의록은 **노션 AI 노트**로 작성하는 것을 우선한다.
- 팀원 전체가 노션 AI 노트를 쓸 수 없으면 **클로바노트**(웹 접속 가능)를 사용한다.

**규칙**

- **회의 시작과 동시에 AI 노트부터 켠다.**
- 다른 사람의 아이디어를 비판하지 않는다.
- 처음 아이데이션 단계에서는 반드시 긍정적으로, 아이디어를 더 확장시킬 수 있는 아이디어만 내놓는다.
- 판단은 보류하고 양으로 승부한다.
- 떠오르면 즉시 기록한다.

**실습**

1. 기본은 **클로바노트**로 아이디어 회의를 진행하고, 그 회의 내용을 문서로 정리한다.
1. **노션 AI 노트**로도 동일하게 실습할 수 있다.

**점검** — 정리된 문서를 공유 → 흥미도·실현성 코멘트

---

## 3. 아이디어 선별

**방법·순서** — 후보 나열 → 후보별 딥리서치로 근거 수집 → 심사 기준 적용 → 우선순위 → 1개 확정

아이데이션에서 모인 후보들을 나열한 뒤, 후보마다 딥리서치를 돌려 근거 자료를 모은다. 딥리서치는 시간이 걸리는 작업이므로 후보가 정해지는 대로 바로바로 돌려두면, 다른 후보를 검토하는 동안 결과를 받아볼 수 있다. 그렇게 쌓인 자료를 심사 기준에 대입해 점수를 매긴다.

**심사 기준 5가지**

1. 문제 정의 및 사회적 필요성
1. 창의성 및 독창성
1. AI 활용의 적절성 및 기술 타당성
1. 실현 가능성 및 완성도
1. 기대 효과 및 파급력

**도구**

- 평가표 — Notion / 스프레드시트
- 딥리서치 — 아이디어별로 아래 도구 중 하나를 선택해 돌린다

| 도구 | 특징 |
|------|------|
| Gemini Deep Research | 긴 리포트 자동 생성, 출처 링크 포함 |
| ChatGPT Deep Research | 웹 검색 기반 단계별 리서치, 근거 풍부 |
| Claude | 긴 문서 분석·요약, 구조화된 평가에 강점 |
| Perplexity | 빠른 팩트체크, 최신 정보 검색에 유리 |

예시 프롬프트는 `Example_prompt/` 폴더 참고.

**규칙**

- 심사위원 경험 및 기준으로 본다.
- "이거 좋은데요?" 반응을 신호로 포착한다.
- 딥리서치는 후보가 정해지는 대로 즉시 돌린다. 결과를 기다리는 동안 다른 후보를 검토한다.
- 리서치 결과와 심사 기준 점수를 함께 노션에 기록한다.
- 정한 이유를 문서에 써둔다.

**실습** — 아이디어 1개를 골라 딥리서치 프롬프트 작성 → 결과를 심사 기준표에 대입 → 후보 점수화 → 1개 선택

**점검** — 선정 사유 발표 → 기준이 명확한지 확인

---

## 4. 문서화

**방법·순서** — 문서 목록 잡기 → 마크다운 작성 → 검토 → `docs/` 폴더에 묶기 → 지속 업데이트

**문서 목록 (예시)**

- 필수 — PRD, TRD, README
- 권장 — 디자인 가이드, 화면·기능 정의, Changelog
- 선택 — 데이터 모델, API 명세, 컨벤션, 배포 가이드, 발표 개요

**도구** — 마크다운 에디터, Claude, Cursor

**규칙**

1. AI에게 너무 의존하면 문장만 늘어난다.
1. 마크다운으로 제작 → AI가 가장 좋아하는 문서 형식
1. 마크다운은 최소 2,000줄 이내로 만들기 → 짧을수록 좋음
1. 문서가 너무 길어지면 주제별로 문서 쪼개기
1. 문서는 프로젝트 루트의 `docs/` 폴더에 묶어두기
1. 의사결정이 필요 → 경험이 많다고 모든 결정이 좋을 수는 없다.

> AI도 결정을 대신 안 한다. 프롬프트를 입력하는 사람의 기반에 맞춰준다.

**실습** — `docs/` 폴더를 만들고 기능명세(PRD)를 마크다운 1장으로 작성

**점검** — "AI가 읽기 좋은 구조인가" 상호 리뷰

## 5. 구현

**방법·순서** — Repo·배포 준비 → `docs/` 문서 확인 → Agent mode로 구현 → (필요 시) Plan mode → 확인 → Commit + Push

1. **Repo·배포 준비** — GitHub 저장소를 만들고 Cursor에서 clone한다. 웹 앱이면 Vercel 등 배포 환경도 연결해 둔다. 구현 전에 작업 공간을 먼저 갖춘다.
1. **`docs/` 폴더 확인** — 4번에서 만든 마크다운 문서(PRD, TRD, README 등)가 `docs/`에 모여 있는지 확인한다.
1. **구현 범위 정하기** — PRD(무엇을 만들지)와 TRD(어떻게 만들지)를 기준으로, 이번에 구현할 기능·화면을 하나 정한다.
1. **구현 지시 (Agent mode)** — `docs/`의 PRD·TRD·README(또는 화면·기능 정의)를 @로 첨부하고, "이 문서 기준으로 ○○ 기능을 구현해줘"라고 지시한다. 한 단위씩 진행한다. 범위·방향이 불명확할 때만 **Plan mode**를 추가로 켜서 계획을 먼저 세우고, 팀에서 확인한 뒤 Agent로 넘어간다.
1. **확인** — 브라우저(또는 배포 URL)에서 `docs/`의 PRD·화면·기능 정의와 실제 동작이 일치하는지 확인한다.
1. **Commit + Push** — 확인된 구현을 GitHub에 올린다. **팀원 누구나** Agent에게 "커밋하고 푸시해줘"라고 지시하거나, Cursor·GitHub에서 직접 처리한다.

> Agent mode가 기본이다. Plan mode는 원할 때만 추가한다. Repo → 배포 → 구현 순으로 잡아도, 오류수정은 6번에서 이어간다.

**4번 문서 → 구현 시 활용**

| 문서 | 구현에서 쓰는 방법 |
|------|-------------------|
| **PRD** | 구현 범위·우선순위·기능 정의의 기준. Plan/Agent 지시의 "무엇을" |
| **TRD** | 기술 스택·구조·제약. Plan/Agent 지시의 "어떻게" |
| **README** | 프로젝트 개요·실행 방법. Agent가 프로젝트 맥락을 파악할 때 |
| **디자인 가이드** (권장) | UI 색·폰트·컴포넌트 스타일 제약 |
| **화면·기능 정의** (권장) | 화면별·기능별 상세. 한 번에 하나씩 구현할 단위 |
| **Changelog** (권장) | `docs/Changelog.md` — 구현·수정 이력. 6번 오류수정에서 기록 |
| **데이터 모델 / API 명세** (선택) | 백엔드·연동 구현 시 @첨부 |

**도구** — Cursor (Agent mode / Plan mode), GitHub, Vercel

- **Cursor (Agent mode)** — 기본. `docs/` 문서 @첨부 후 구현 지시. Plan mode는 원할 때만 추가.
- **GitHub** — 코드 저장·버전 관리. Commit + Push. 팀원 전원이 할 줄 알아야 한다.
- **Vercel** — 웹 앱 배포(선택)

**규칙 — 문서 기반으로 잘 시키는 법**

4번 규칙(마크다운, `docs/` 폴더, 짧게·쪼개기)을 전제로, 구현 지시문을 쓴다.

1. **문서를 먼저 붙인다** — `docs/`의 PRD·TRD 등 마크다운을 @로 첨부하거나, "`docs/PRD.md` 기준으로"처럼 경로를 명시한다. 맥락 없이 "로그인 만들어줘"만 하지 않는다.
1. **문서 종류별로 역할을 나눈다** — PRD로 범위, TRD로 기술·제약, 화면·기능 정의로 단위. 한 지시문에 모든 문서를 섞지 말고, 이번 작업에 필요한 것만 골라 준다.
1. **한 번에 하나씩** — PRD의 기능·화면 정의 단위(또는 화면·기능 정의 1개)마다 Agent mode로 구현한다. 방향이 불명확할 때만 Plan mode를 먼저 쓴다.
1. **제약을 문서에서 가져온다** — TRD의 스택·구조, 디자인 가이드의 UI 규칙을 지시문에 그대로 적는다. AI가 문서와 다른 선택을 하지 않게 한다.
1. **기대 결과를 문서로 보여준다** — 화면·기능 정의의 설명, PRD의 acceptance criteria, 예시 UI 문구 등을 지시문에 인용한다.
1. **문서와 코드가 어긋나면 문서를 먼저 본다** — 구현 결과가 PRD/TRD와 다르면, 지시문만 고치기 전에 "문서가 맞는지" 팀에서 확인한다. (4번 규칙: AI는 결정을 대신하지 않는다.)
1. **확인 후 Commit + Push** — `docs/` 기준으로 동작이 맞으면 GitHub에 올린다. 개발자에게만 맡기지 않는다.
1. **안 되면 다시** — 결과를 실행·확인하고, 부족하면 첨부 문서·지시문을 고쳐 반복한다.

> 구현 품질은 **4번 문서의 명확함**과 **5번 지시문이 `docs/` 문서를 얼마나 정확히 참조하는지**가 함께 결정한다.

**실습** — GitHub Repo 생성·Cursor clone → `docs/`의 PRD(및 TRD)를 @첨부 → Agent mode로 첫 기능 1개 구현 → 브라우저에서 확인 → Commit + Push

**점검** — 구현 결과가 `docs/`의 PRD·TRD·화면·기능 정의와 일치하는지 대조 → GitHub에 커밋이 올라갔는지 확인 → 어긋난 부분은 문서 수정 vs 지시문 수정 중 어디를 고칠지 팀에서 결정

## 6. 오류수정

**방법·순서** — 문제 발견·기록 → 재현 → `docs/`와 대조 → Agent mode로 수정 → 확인 → `docs/Changelog` 기록 → Commit + Push

오류는 반드시 생긴다. 비개발자도 **코드를 직접 고칠 필요는 없다.** 핵심은 **무엇이 잘못됐는지 정확히 적고, AI에게 잘 시키고, 결과를 눈으로 확인하는 것**이다. 수정이 끝나면 **Commit + Push도 직접** 한다. 개발자만의 일이 아니다.

1. **문제 발견·기록** — "어떤 화면에서 / 어떤 순서로 / 무엇을 했더니 / 어떤 결과가 나왔는지"를 적는다. 스크린샷·화면 녹화를 붙이면 더 좋다.
1. **재현** — 같은 순서로 2~3번 다시 해본다. 매번 같은 증상이면 "재현 가능한 오류"다.
1. **`docs/`와 대조** — PRD·화면·기능 정의에 적힌 **의도한 동작**과 비교한다. "버그인지, 문서가 틀린지, 처음부터 다른 기획인지"를 먼저 구분한다.
1. **수정 지시 (Agent mode)** — 재현 방법 + 기대 결과 + `docs/` 관련 문서를 @첨부하고 Cursor Agent에게 수정을 지시한다. (5번 구현 지시와 같은 방식) 원인·방향이 불명확할 때만 **Plan mode**를 추가로 켜서 계획을 먼저 세운다.
1. **확인** — 브라우저(또는 배포 URL)에서 **직접 클릭·입력**하며 고쳐졌는지 확인한다. 처음 문제 났던 순서 그대로 다시 테스트한다.
1. **기록** — `docs/Changelog.md`에 "무엇이 / 왜 / 어떻게" 고쳐졌는지 남긴다. (4번 문서 목록의 Changelog)
1. **Commit + Push** — 확인된 수정을 GitHub에 올린다. **팀원 누구나** Agent에게 "커밋하고 푸시해줘"라고 지시하거나, Cursor·GitHub에서 직접 처리한다.

> 비개발자의 오류수정 = **발견 · 재현 · 기록 · 확인 · Commit + Push**. 코드 수정은 Agent mode, Plan은 필요할 때만.

**비개발자 / 개발자 역할**

| 역할 | 하는 일 |
|------|---------|
| **비개발자 (주도)** | 문제 발견·재현, `docs/` 대조, Agent mode 수정 지시, 브라우저에서 결과 확인, `docs/Changelog.md` 작성, **Commit + Push** |
| **개발자 (보조)** | Agent 지시 보완, 개발자도구로 에러 로그 확인(필요할 때), 배포 상태 점검 |

**도구** — Cursor (Agent mode / Plan mode), 브라우저, GitHub, `docs/` (PRD, 화면·기능 정의, Changelog)

- **Cursor (Agent mode)** — 기본. `docs/` 문서 @첨부 후 수정 지시. Plan mode는 원할 때만 추가.
- **브라우저** — 배포 URL 또는 로컬 미리보기에서 직접 확인. 비개발자의 주된 검증 도구.
- **GitHub** — Commit + Push. 팀원 전원이 할 줄 알아야 한다.
- **`docs/Changelog.md`** — 수정 이력. 4번 문서 목록(권장)과 동일. 7번 발표·시연 자료로 이어진다.
- (선택) **브라우저 개발자도구** — 개발자가 콘솔 에러·네트워크 오류 확인할 때. 비개발자는 필수 아님.

**규칙 — 비개발자 오류 수정 지시문**

5번 규칙을 그대로 이어 받는다. 오류 수정도 **문서 기반 지시**다.

1. **증상을 구체적으로** — "안 돼요" 대신 "로그인 버튼 클릭 → 3초 후 흰 화면"처럼 적는다.
1. **기대 결과를 문서에서 가져온다** — "`docs/PRD.md`의 ○○ 기능 기준, ○○가 되어야 한다"고 명시한다.
1. **재현 순서를 그대로 적는다** — 1단계, 2단계… 클릭·입력 내용까지. AI가 같은 상황을 만들 수 있게 한다.
1. **한 번에 하나만** — 오류 1건당 지시 1개. 여러 문제를 한 프롬프트에 섞지 않는다.
1. **스크린샷·녹화를 붙인다** — 말로 설명하기 어려운 UI 문제는 캡처가 지시문보다 정확하다.
1. **고쳐졌는지 직접 확인한다** — AI가 "수정했습니다"라고 해도, 브라우저에서 재현 순서대로 다시 테스트한다.
1. **확인 후 Commit + Push** — 본인이 원하는 동작이 구현됐을 때 기록하고 GitHub에 올린다. 개발자에게만 맡기지 않는다.
1. **새 기능 추가와 구분한다** — "고치기"와 "기능 추가"는 다르다. 기능 추가는 **New 세션**을 열고 5번 구현 흐름으로 진행한다.

**실습** — 의도적으로 만든 오류 1건(예: 버튼 클릭 시 반응 없음)을 재현 → `docs/PRD.md` @첨부 → Agent mode로 수정 지시 → 브라우저에서 확인 → `docs/Changelog.md`에 3줄 기록 → Commit + Push

**점검** — `docs/Changelog.md`에 "증상 / 원인(추정) / 해결"이 적혀 있는지, PRD·화면·기능 정의와 실제 동작이 일치하는지, GitHub에 커밋이 올라갔는지 확인

## 7. 발표준비

**방법·순서** — 목차 → 내용 → 시연하며 슬라이드 생성

**도구** — Gemini, NotebookLM, Claude, Claude Design

**규칙**

- 문서는 이미 만들어져 있다. 구현·시연은 재미있게 만들어가는 게 낫다.
- 보통 발표 내용은 AI로 만들면 된다.
- 슬라이드 생성은 쉽다. 안에 들어가는 내용이 항상 어렵다.
- 한 번에 모든 슬라이드를 만드는 것보다 목차 → 내용 → 시연하며 생성.

**실습** — 문서 기반으로 3장짜리 슬라이드 초안 생성

**점검** — 모의 발표 → 시간·전달력 확인

## 자주 묻는 질문

**Q. 해커톤 하느라 바빠 죽겠는데 소통 창구 만들 시간이 어딨어요?**

바쁠수록 더 필요하다. 소통 창구가 없으면 "아까 누가 뭐 말했더라"를 반복하다 시간을 다 쓴다. 노션 페이지 만들고 팀원 초대하는 건 5분이다. 그 5분이 이후 문서화 → AI 구현 → 발표준비까지 전부 연결되는 기반이 된다.

**Q. 비개발자는 해커톤에서 뭘 하나요?**

할 일이 더 많아진다. 이제는 직접 코딩하는 것보다 **AI에게 잘 시키는 것**이 핵심이다. 비개발자는 문제 정의, 자료 조사, 딥리서치, 문서화(PRD·기획서), 지시문 작성, **오류 수정(Agent mode)**, **Commit + Push**, 발표 준비에서 큰 역할을 한다. 오히려 "무엇을 왜 만드는가"를 정리하는 일은 비개발자가 더 잘하는 경우가 많다.

**Q. 노션 말고 다른 툴을 써도 되나요?**

된다. 중요한 건 툴이 아니라 **모든 기록을 한곳에 모으는 것**이다. 다만 이 강의는 나중에 MCP로 노션 기록을 AI 컨텍스트로 끌어다 쓰는 흐름을 전제로 한다. 팀이 이미 익숙한 협업 툴이 있다면 그걸 써도 되지만, 기록이 여러 곳에 흩어지지 않도록 한 곳을 기준으로 정하자.

**Q. AI가 만든 코드나 문서를 그대로 믿어도 되나요?**

안 된다. AI는 결정을 대신해주지 않고, 지시한 사람의 기반에 맞춰 결과를 낼 뿐이다. 코드는 반드시 직접 실행·테스트해 확인하고, 문서·리서치 결과는 출처와 사실 여부를 검토한다. 특히 딥리서치 결과의 통계·근거는 한 번 더 교차 확인하자.

**Q. 딥리서치 결과가 너무 길고 많은데 어떻게 정리하나요?**

전부 읽으려 하지 말고 **심사 기준 5가지에 대입**해 필요한 부분만 추린다. 원본 리포트는 노션에 `자료`로 저장해두고, 핵심만 뽑아 심사 기준표와 함께 기록한다. 발표 문구로 옮길 땐 정리된 핵심만 가져다 쓰면 된다.

**Q. 팀원이 AI를 한 번도 안 써봤어요. 어디서부터 시작하죠?**

회의 때 **AI 노트(노션 AI 노트 또는 클로바노트) 켜기**부터 시작한다. 말만 하면 기록이 남는 경험이 가장 진입장벽이 낮다. 익숙해지면 검색·딥리서치·지시문 작성으로 자연스럽게 넘어간다. 처음부터 모두가 능숙할 필요는 없고, 역할을 나눠 잘하는 사람이 먼저 시도하면 된다.

**Q. 비개발자도 오류 수정과 Commit + Push를 하나요?**

한다. 코드를 직접 고치는 게 아니라, **문제를 재현하고 기록하고, `docs/` 기준으로 Agent mode에 수정을 지시하고, 브라우저에서 확인**하는 역할이다. 수정이 확인되면 **`docs/Changelog.md`에 기록하고 Commit + Push까지** 한다. 개발자만의 일이 아니며, 해커톤 팀원 **누구나** 할 줄 알아야 한다.
